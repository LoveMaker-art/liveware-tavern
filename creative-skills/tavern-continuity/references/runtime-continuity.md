# Runtime Continuity Model

## Confirmed History

The latest user turn remains raw and unconfirmed. The runtime compresses only completed user/assistant turns.

## Fifteen-Turn Checkpoint

- STORY_STATE_BATCH_TURNS defaults to 15.
- Each successful batch advances the plot-ledger checkpoint and runtime-cast checkpoint atomically.
- A failed ledger or cast update advances neither checkpoint.
- The same batch is retried later after failure.
- Backlogged stories are processed in complete chronological 15-turn batches.

## Plot Ledger Ownership

The story ledger owns:

- current scene, locations, participants, and activities;
- completed and ongoing events;
- causal facts and knowledge boundaries;
- key objects and custody;
- secrets, promises, conflicts, and open threads.

The ledger is bounded by STORY_STATE_MAX_CHARS, currently defaulting to 15000 characters. Size reduction removes duplicates, expired scene detail, and inconsequential completed actions before protected memory.

When a valid ledger covers confirmed turns, those covered raw messages are replaced in model context by the ledger. Uncovered recent turns remain raw. Frontend story display remains unchanged.

## Runtime Cast Ownership

runtime_cast is the sole world-local authority for effective character and user profiles.

- origin_profile and origin_user_profile are immutable entry snapshots.
- profile and user_profile are the only effective profiles.
- Persistent status and relationships contain durable changes only.
- Temporary emotion, action, location, clothing, short-term goal, knowledge, and held objects stay in the story ledger.

Character evolution uses only the same newly confirmed 15-turn batch as the ledger. It must never replay the full story.

## Rebuild Rule

If the plot ledger is invalid or explicitly rebuilt:

1. Rebuild only the ledger in chronological batches.
2. Preserve current effective profiles, persistent statuses, and relationships.
3. Move the preserved cast checkpoint to the rebuilt ledger boundary.
4. Do not call the character-state model during ledger rebuild.

## Turn Planning

Multi-character per-turn scheduling lives in `/opt/data/apps/tavern-runtime/turn_plan_service.py`. It builds the short `turn_plan` for multi-character replies, using the current cast, effective story_state, response language, and recent raw story excerpt. The runtime caller provides the active model/chat function and JSON parser. Failure returns `{}` so generation can continue. A foreground turn should generate one plan and reuse it for empty-reply retry.

## Reply Formatting

Actor reply normalization lives in `/opt/data/apps/tavern-runtime/reply_format.py`. It owns quote normalization, role-label colon normalization, pure-narration `*...*` wrapping, and the continue-story direction text. It is pure formatting: no model calls, no state writes, and no production access.

## Message Segments

New `char` messages may carry a hidden `segments` helper index generated by runtime rules in `/opt/data/apps/tavern-runtime/message_segments.py`, not by an extra model call.

Fields:

- `speakers`: characters with explicit or safely inferred dialogue.
- `dialogue_blocks`: `{speaker, text}` blocks; inferred blocks include `inferred: true`.
- `narration_blocks`: narration/action paragraphs with mentioned character names.
- `action_only`: characters present in narration but not recognized as speakers.
- `mentioned`: current cast names found anywhere in the reply.

Parsing rules are intentionally conservative: explicit `角色名：「对白」` is authoritative; standalone `「对白」` is attributed only when the previous narration paragraph mentions exactly one cast member. `segments` is an auxiliary index for diagnosis, TTS, and future scheduling analysis. It must not directly mutate `story_state` or `runtime_cast`; raw message `text` remains authoritative.

## Generation Service

Generation orchestration lives in `/opt/data/apps/tavern-runtime/generation_service.py`. It owns the runtime loadout and actor-generation entrypoints used by `send_message`, `continue`, `regenerate`, swipe, and edit-continue flows.

Responsibilities:

- load cards, worldbooks, persona, story, and author note for a production;
- call `actor.perform` with the active model, effective `story_state`, response language, and optional `turn_plan`;
- preserve the one-turn-plan-per-foreground-turn rule by accepting a caller-provided `turn_plan`;
- route empty-reply retry through the same generation service while applying reply normalization at the server boundary.

It should remain orchestration-only: no direct persistence writes, no compression, no cast mutation, and no hidden frontend behavior. `server.py` keeps route/event ownership and wraps this module with runtime dependencies such as the active model and language resolver.

## Story State Service

Deterministic plot-ledger helpers live in `/opt/data/apps/tavern-runtime/story_state_service.py`. This module owns turn counting, confirmed-turn compression boundaries, batch rendering/splitting, cheap token estimation, ledger normalization/trimming, and deterministic story-ledger shape/reference validation.

It must stay model-free and persistence-free. `server.py` still owns validated model calls, background scheduling, locks, atomic publish, and event routing. This keeps the compression pipeline inspectable without changing when or how state is committed.

## Runtime Cast Service

Runtime-cast deterministic validation and snapshot application live in `/opt/data/apps/tavern-runtime/runtime_cast_service.py`. This module owns the cast-update contract prompt, compact-output canonicalization, evidence validation, audit coverage checks, no-op rejection, and applying accepted durable profile/status/relationship diffs to a new `runtime_cast` snapshot.

It must stay persistence-free and scheduler-free. `server.py` still owns the memory-model call/retry loop, atomic ledger+cast publish, production locks, and background job routing. This keeps character-state evolution reviewable while preserving the existing 15-turn checkpoint semantics.

## Runtime Regression Tests

Core runtime service regressions live in `/opt/data/apps/tavern-runtime/tests/test_runtime_services.py`. They use Python stdlib `unittest` and fake model/actor callables, so they do not call the real model or mutate production state.

Run them with:

```bash
cd /opt/data/apps/tavern-runtime
/opt/hermes/.venv/bin/python -m unittest tests.test_runtime_services -v
```

The suite currently covers message segmentation, reply formatting, turn-plan success/failure behavior, empty-reply retry reusing the supplied `turn_plan`, story-ledger deterministic helpers, and runtime-cast evidence/audit/application rules. Run this before and after changing continuity prompts, generation orchestration, or repair flows.

## Diagnostic Checks

Compare:

- compressible confirmed turns;
- story_state.turns;
- runtime_cast.applied_turn;
- runtime_cast revision and last error;
- ledger size and required fields;
- raw messages actually selected by actor.py;
- prompt size and model response timing.

The story_state and runtime_cast checkpoints should match after every successful batch.
