> 这是**一起想**的工作稿，不是定案。基于 `sillytavern-ecosystem.md` 的事实展开。

## 0. 一句话产品命题（先讲产品，再讲技术）

**SillyTavern 是一个「没有 agent 的前端」；ClawChat 是一个「能托管前端的 agent 平台」。**

ST 的全部复杂度，本质来自一个缺失：它没有一个持久的、有身份和记忆的主体。所以它必须让用户每次手工**搭台**——拼 prompt、调 preset、喂世界书、维护 summary——用一套静态的「角色卡 + 提示词工程」去**模拟**一个角色。

ClawChat 反过来：agent 本身就是持久主体（联系人、有记忆、能调能力服务器、能托管 liveware）。所以我们不需要让用户搭台——**角色可以直接是一个 agent 联系人**，记忆和人格演化是 agent 的原生能力，而 liveware 提供聊天气泡给不了的「沉浸面」。

这就是「更好的酒馆体验」的支点：不是把 ST 抄一遍做得更顺手，而是用**「活的角色」**取代**「静态角色卡 + 手工搭台」**。

## 0.5 已拍板的产品愿景（2026-06-28，用户定）

### 定位：工具视角，内容中立
**不在「NSFW 与否」的内容轴上站队**。我们做的是 agent IM 平台上的**角色扮演工具**——内容是用户和 agent 之间的事，平台提供工具、不编辑不预设立场。这与 ClawChat 一贯的平台/工具姿态一致，也绕开上架/品牌的内容纠结。原"岔路 1"作废。

### 形态：搭子 + 控制台
- **agent = 角色扮演搭子（一个会入戏的演员）**：不是一堆静态卡，而是一个能即刻"变成"所选角色的伙伴。
- **liveware = 角色扮演控制台**：管理**角色卡库 / 世界书库 / 对话线**，并负责**切换**。
- **核心闭环**：控制台里切角色 + 世界书 → agent 即刻进入对应聊天状态 → 用户回 IM 接着聊。= 随手换搭子、换剧本。

### 心智模型：active loadout（当前装载）
agent 始终有一份"当前装载" = `(角色卡, 世界书, 对话线)`。控制台 = loadout 管理器；切换 = **加载**某套 loadout，agent 据此入戏。

- **数据归属**：角色卡 / 世界书 / 对话都是**用户的端侧权威资产**（呼应 ClawChat 端侧存储、无服务器删除），控制台读写、agent 读取当前 loadout 作为上下文。
- **切换语义（已确认）**：每套 `(角色, 世界书)` 组合带**自己的独立故事线**；切换 = 跳进那条线的现场（像微信里切不同会话），而非同一对话里换装。详见下「演员模型」。

### 演员模型（2026-06-28 深化）

**底色 = 一个演员，可同时跨多个剧组、各演各的、各自融进对应故事线。**

- **剧组（production）** = 一套 `(角色, 世界书, 独立故事线[+ 可选其他角色])`。演员在某剧组"在场"时，全程入戏、只带该剧组的连续性。
- **演员（agent）** = **唯一身份**，同时挂在多个剧组上，像演员周一拍 A 剧、周二拍 B 剧。

这一锁，自动定下五件事：
1. **单一 agent 身份**——不是"每张角色卡一个 agent"，是一个演员演 N 个角色。用户的关系对象是**这个搭子**。
2. **各剧组记忆隔离 = 一级需求**——绝不把 A 剧的记忆漏进 B 剧的场（ST 用户手工防"串味"的痛点，这里做成结构保证）。
3. **liveware = 剧组管理台 / 后台**——管剧组列表、切换在场、开新戏、归档；用户说的"对话管理"即此。
4. **角色卡库 = 选角库**——导入一张卡 = 给演员一个可演的新角色。Chub 6 万+ 卡 = 现成选角池。
5. **群聊 RP 自然外延**——一个剧组里多角色（一人分饰/多演员同台）顺势可接，暂缓。

### 底色自我 = A（已定 2026-06-28）+ 核心差异化

演员有一个**贯穿各剧组的持久自我**（你的搭子）：能"出戏"和 ta 本人聊、跨剧组记得你的偏好、推荐新戏、共创。**这是我们和酒馆最大的区别。**

**角色 = 好演员 × 好剧本（协作产物）。** 剧本（角色卡）可导入（Chub 生态），但它只是**一半**；演员带来诠释，且会**成长**。同一张卡，被"懂你的演员"演 vs 被通用引擎跑，是两个角色——ST 只给剧本 + 可替换引擎，给不了演员。

**这逼出一个关键架构：两层记忆，正交。**

| 层 | 范围 | 装什么 | 规则 |
|---|---|---|---|
| **故事记忆** | per 剧组（隔离） | 剧情事实、这条线发生了什么 | 绝不串味，留在本场 |
| **演员技艺 / 对你的了解** | 跨剧组（共享） | 你的互动偏好（节奏/语气/主动性/人称/边界）+ 演员自己的签名风格 | 带进每个片场，但**不带剧情** |

正是这一层正交解决了"**搭子懂你**"与"**不串味**"的张力：**剧情隔离、技艺共享**。像真演员把自己的功力和对你的了解带进每个角色，却不会把 A 角色的台词说进 B 角色的戏。

**为什么这是护城河（ST 结构上做不到）**：
- **演员独特性 = 不可替代**：你的演员被你"调教"得越来越懂你，越演越好。ST 一切可移植 / 一次性，没有关系沉淀、没有留存。
- **资产归属**：演员的"技艺画像"极私密 → 端侧权威、用户所有。**剧本可移植，关系不可移植**——别人下载不走"你的演员"。
- **回应 ST issue #2022（角色不可演化）**：这里角色通过"**演员诠释 + 与你共同打磨剧本**"真实演化——ST 想做做不到的事，在我们这是自然结果。

### 可行性裁决（2026-06-28 已核，绿灯）

完整裁决见 [feasibility-liveware-agent-channel.md](feasibility-liveware-agent-channel.md)。结论：

- **核心闭环能用 ClawChat 现有管线搭起，不需要把容器桥提前。** 四跳：① 控制台→agent server（同源 `fetch('/api/event')`，域锁只拦顶层导航、放行同源 fetch，digest 已验）② agent server 存 loadout（state 文件，仿 digest `subscriptions.json`）③ agent→IM 主动开场（gateway 进程内 emit `message.send`，协议无前序消息门控）④ 用户↔agent 聊 + 读 loadout（既有 1:1 会话）。①②④ **现成可用**，③ **需补但非桥**——靠 digest 同款 cron/webhook 唤醒 hack。
- **影响面**：①②③④ 核心逻辑全是**纯 agent 侧 skill**（独立仓，不碰 ClawChat）；**ClawChat 客户端零必改**；**零新增后端接口（铁律 B 不触发）**。红线：active loadout 只落 agent 侧 state，别写进能力服务器/member-backend（写了就触发 B）。
- **v1 头号风险**：③ "切角色后多久开场"的触发即时性（cron 定时 / webhook 未坐实，秒级 vs 分钟级待验）。
- **两个新浮现的 v1 设计真空**（recon 挖出，下方「v1 待解」）：角色身份呈现 + 多故事线↔单 IM 会话的映射。

## 1. 三个高度，是层不是互斥

| 层 | 命题 | 类比 |
|---|---|---|
| **A. 角色即 agent 联系人** | 角色卡导入 → 变成你 IM 里的 agent 联系人，记忆/演化是原生能力 | 产品的"魂" |
| **B. 酒馆活件（沉浸面）** | liveware 容器承载聊天气泡给不了的富交互：状态栏 / 立绘 / 场景图 / 世界书 / 分支选择 / 跑团工具 | 产品的"身" |
| **C. 酒馆能力服务器** | 类 newsdesk 的能力底座：角色卡解析、世界书 RAG、出图(SD/Comfy)、TTS、记忆库、发现/分发 | 产品的"骨" |

真正的产品 = **A 当魂、B 当身、C 当骨**，分阶段长出来。下面逐层说清"agent+liveware 比 ST 强在哪"。

### A. 角色即 agent 联系人 —— 直击 ST 两个最硬的结构性空白
- **打掉「搭台」**：ST 把 prompt 拼装、preset、上下文模板暴露给用户（21 个采样项的"设置海"）。agent 平台把推理 + 身份 + 记忆全接管，用户**像和人聊天一样和角色聊天**。直接命中 ST 第一痛点（上手门槛）。
- **角色会演化**：ST 官方 issue #2022 被 stale 关闭——"Character X will always be character X"，角色卡是静态定义。agent 原生记忆 = 角色**会记住、会变化、会沿剧情长**。这是 ST 结构上做不到的差异化（机会信号 #4）。
- **记忆自动化而非暴露**：ST 的 summarize+向量+作者注释+记忆表是"多层手工堆叠"。我们把这套分层（短期摘要 / 长期向量 / 作用域隔离）做成 agent 后台自动跑，用户不感知（信号 #5）。
- **代价 / 张力**：ST 重度玩家**享受**调参搭台（控制欲是乐趣的一部分）。全自动会"夺走玩具"。→ 用 B 层的"角色工坊"活件把控制权以**可选**形式还回去（想精调的人能精调，不想的人开箱即玩）。

### B. 酒馆活件 —— liveware 把"聊天"升级成"沉浸"
聊天气泡只能承载文字流。酒馆的沉浸感来自 ST 用户在 Tavern Helper 里**自己搭**的那些东西，liveware 可以原生给：
- **角色状态面板**：好感度 / 心情 / HP / 物品栏 / 关系网——即社区"动态状态栏"玩法，agent 驱动数据、liveware 渲染。
- **场景与立绘**：生成场景图、随情绪切换的角色立绘 / Live2D（对应 ST 的 expressions/sprites）。
- **世界书浏览/编辑器**：把 lorebook 从"手填关键词的负担"变成可视化、可被 agent 主动维护的面板。
- **分支选择 / 视觉小说式叙事**：在富界面里给选项、走剧情树。
- **跑团 / GM 工具**：掷骰、地图、角色卡表——对应 ST 的文字冒险玩法。
- **关键架构优势（护城河）**：Tavern Helper 证明了"agent 产出活件、客户端渲染交互 UI"的**强需求**，但它**无沙箱、注入即全权限、2026-05 刚出过窃 API key 事故**（信号 #6）。liveware 容器是**进程级隔离 + 域锁 + 不注入 token + 能力默认关**——同样的体验，安全边界根本不同。我们能正大光明地让第三方做"酒馆活件"，ST 不能。

### C. 酒馆能力服务器 —— 把生态的硬资产变成服务
类 newsdesk 的独立能力底座，agent 经 skill 调用：
- **角色卡解析/导入**：吃 V2/V3 PNG（`chara`/`ccv3` chunk）+ CHARX(zip)，转成 agent 人设 + 记忆种子。V2 是最大公约数，**供给极充足**（Chub 6 万+ 卡）。
- **世界书 RAG**：lorebook 条目进向量库，agent 按需检索而非靠关键词硬触发。
- **出图 / TTS**：代理 SD/ComfyUI、TTS（对接现有或自建）。
- **发现 / 分发 / 版本**：ST 角色卡靠裸 PNG 手传、易被剥元数据、无注册中心（信号 #8）。能力服务器 + liveware 注册体系可成"带质量信号且可演化"的下一代资产层。
- **边界教训**：ST-Extras"外置 Python 边车 → 内化进主程序"的废弃史（信号 #9），提醒我们想清楚"哪些能力内嵌、哪些外挂"。

## 2. 复用 vs 重建：技术取舍已经清楚

- **复用数据格式，不复用引擎**。角色卡 V2/V3 + 世界书是**跨前端事实标准**、文件可独立流通、`extensions` 字段「未知 key 不可删」+ 命名空间化——这是为"多客户端共享同一载体"量身设计的（信号 #3）。我们**吃这个格式**（导入存量 6 万+ 卡），但**不嵌 ST**：ST 无 headless、维护者明确拒绝"当后端被驱动"（#3518），其 prompt 引擎绑死浏览器运行时。
- **范式参考，不接管原语**。STscript / Quick Replies / registerFunctionTool 给我们"楼层内自动化"的设计参考，但都跑在 ST 前端里，复用价值在"看它怎么想"，不在"接进容器"。
- **结论**：吃它的**内容资产**（卡 + 世界书），学它的**交互范式**（状态栏 / 立绘 / 楼层 UI），用我们的 **agent + liveware 架构**重做"魂和身"。不 fork、不内嵌 ST。

## 3. 必须先拍板的战略岔路（这些是产品决策，不是技术）

### 岔路 1 —— 内容定位（**最高优先级，gate 一切**）
酒馆生态的引力极大一部分来自**无审查 NSFW + 越狱**。但 ClawChat 是要上架、做社区发布、有 OpenClaw/Hermes turnkey 的真实产品。这两者直接冲突。
- 选项：① 明确**不碰** NSFW，吃"陪伴 / 互动小说 / 跑团 / 协作写作"这块**干净但同样巨大**的需求面，用 agent 质量 + 记忆 + 富 liveware 取胜；② 走灰色地带（基本和 ClawChat 产品路线、上架、品牌冲突，不建议）。
- **我的判断**：走 ①。酒馆用户的"出逃"动机不止 NSFW——可控、隐私、紧跟模型、角色会成长，这些**干净诉求**ClawChat 全占。但这是你的决定，因为它决定我们对外怎么讲、吸引谁。

### 岔路 2 —— 战略重心：A（魂）优先 还是 B（身）优先
- **A 优先**（角色即 agent 联系人）：最 ClawChat-native、差异化最大、最攻 ST 软肋；但离"ST 用户熟悉的样子"最远，要教育市场。
- **B 优先**（先做一个酒馆活件壳）：对 ST 用户最熟悉、能直接秀"富沉浸面"、demo 最快；但容易变成"在 ST 主场拼 UI 丰富度"，丢掉我们的结构优势。
- **我倾向**：**A 当北极星、B 做第一个可玩 demo** —— 用一个最小"角色活件"（导入一张卡 → agent 联系人 + 一个状态栏/立绘面板）同时验证魂和身，C 先用最薄的解析能力顶上。

### 岔路 3 —— 第一个具体产物是什么
候选：① 角色卡导入器（吃一张 Chub 卡变 agent 人设，验证 C 的解析 + A 的落地）；② 一个"角色主页"活件（状态栏 + 立绘，验证 B 的渲染 + agent 双向驱动）；③ 一篇定位/可行性 memo（先不写码，把战略对齐）。

### RP 主舞台 = 混合（2026-06-28 定）

- **IM = 演员/搭子面**：出戏、meta、跨剧组关系、提醒、快聊。一条会话即可，因为它对应"一个持久演员"。
- **控制台(liveware) = 剧组面**：每个剧组各有**独立、分明、入戏**的对话视图，富面（立绘/状态栏/世界书）也长在这里。
- **关键洞察：两面分工 = 两层记忆的 UI 映射。** IM ↔ 演员技艺/对你的了解层（共享）；控制台各剧组 ↔ 故事记忆层（隔离）。**界面就是记忆模型的具象。**
- **顺带 de-risk（重要）**：深度 RP 对话搬进 **控制台 ↔ agent server（同源 request/response，全绿、digest 已证）**，于是有风险的 ③「agent 主动推 IM」**退出关键路径**——v1 核心不依赖它，降级成"演员在 IM 主动冒泡"的可选润色。
- **身份呈现随之清掉大半**：角色身份在控制台里**自由渲染**（web UI，名字/头像/立绘随便上）；IM 联系人保持"演员"固定身份（一个 profile 正好够）。

**v1 待解（缩小后）**
