# 演员卡 actor-card（v1 spec）

> **承接**：`liveware-frontend.md`（活件前端设计语言——`ac*` 前缀、token、原则都来自那里）+ `surfaces-and-features.md §活件特性可见`（养成感 = 可见特性）+ `v1-build-spec.md`（技艺层 `actor_self.md` 数据）。
> **这份定**：演员卡这个 surface 有哪些字段、数值怎么算、亲密度怎么定级、`ac*` 组件长啥样、读路由契约。
> **状态**：2026-07-01 起。

---

## 0. 是什么 / 为什么

**养成感 surface**。把"演员养成"做成一张**可端详的卡**——镜像角色卡的形，但翻到**演员**这一面。**ST 有会用完的角色卡，我们有会长的演员卡** = 差异化的具象。

- **独立活件卡入口**：活件入口看到两张卡（`console` 剧组管理台 + `actor-card` 演员卡）。**各注册为一个独立 liveware app**（`墨的酒馆` + `墨的演员卡`），但由**同一个 `server.py` 服务、共享 `console.css`**——server 靠 relay 透传的 `X-Forwarded-Host` 在 `/` 上分流到 `actor.html` vs `index.html`（演员卡 app 域名存 `state/actor_host.txt`；机制详见 `liveware-frontend.md §5`）。墨拿链接走 `tavern_cli.py card`（`/api/actor_card` 的 `actor_url`）。
- **零新后端**（铁律 B 不触发）：全是对**现有 state 的聚合读**，不加后端字段、不碰 member-backend。
- **进阶线 MVP 的体验半**：养成的"管道半"（自动累积 + 合并）落在 `actor_self.md`（§7），这张卡把它**变得看得见、被感到**。

## 1. 数据来源（全现成）

| 用到的 | 来自 |
|---|---|
| 出道日 / 剧组数 | `productions/*.json` 的 `created_at`、文件数 |
| 演出轮数 / 演出量字数 / 活跃时间 | 各 production `story[]` 的 `role`（数用户出手）、`text`（累加字数）、`ts` |

| tagline / 我对你的了解 / 生涯年表 | `actor_self.md` 的 **底色** / **我对你的了解** / **成长记** 三段 |

## 2. 字段 + 数值算法

**读法**：标 **v1** 的进第一版；**v1.1** 留迭代（`console.css` 里 `ac*` 类先占位、数据后接）。

### 身份头（v1）
`✦ 墨 · 演员` + 一句 tagline。**tagline 提取**：取 `actor_self.md` 底色段里"演员——"破折号后那句（种子 = "能钻进任何角色里的人"）；**别用"第一句"**（那是"我叫墨"）。建议种子在底色显式保留这句短语，解析取它。✦ = `--brand`，是**不可换的品牌锚**（肖像未来由墨自己定，见 `liveware-frontend.md §7`）。

### 生涯数值（v1）
一排数值格（`.acStat` = 大数 + 小标）：

| 数值 | 算法 |
|---|---|
| **出道 N 天** | `今天 - min(所有 production.created_at)`，天 |
| **P 个剧组** | production 文件数 |
| **T 轮演出** | Σ 各 production 中 `role=="user"` 的条数（= 总出手数，和 reader 剧目卡"N 轮"同口径） |
| **Y 万字** | Σ 各 story 中**非 user 侧** `text` 字符数（演出量，≈ token，估算）。**排除每个剧组的 `story[0]` 开场白**（first_mes 是卡作者写的，非墨生成）；**只算 active 成品**，不含重生成的 `alts`（成品口径，非消耗口径） |
| **戏路 Z**（演过 Z 个角色） | 去重的 `card_id` 数。**不叫"搭档"**——搭档 = 你（对手戏那个人，见 §3 亲密度）；墨演的凛/林夏是**角色/戏路**，不是搭档（词汇 canon，`liveware-frontend.md §术语表`） |

### 亲密度 · 与你（v1）——养成感的情绪核心
一个复合分驱动的**等级**（§3 阶梯）+ 进度条 + 一句人话：

```
亲密度分 = 总演出轮数 T  +  8 × 生涯年表条数 L
```
（`L` = `actor_self.md`「成长记」段的**累计**条目数 = 生涯年表长度。阈值/系数是 env 可调常量 `ACTOR_INTIMACY_*`，二创可调不动结构。）

**为什么是年表 L、不是「我对你的了解」条数**：Q_B 之后「我对你的了解」是**合并、有界**的（~10 条封顶，合并时甚至会减少）——拿它驱动一个只该涨的亲密度会**卡住/倒退**。「成长记」append-only、只增，才是"我们一起走过多少"的真信号。

显示：`默契 · 一起 128 轮 · 记下你 12 笔`（"12 笔" = 年表条数 L；**别写"口味"**——口味 = 「我对你的了解」那栏，是另一个数）。**为什么这两个变量**：演得多（陪伴）× 一路调整得多（越来越懂）—— 正好是"持久搭子"的两个轴，不是单纯刷时长。

### 我对你的了解（v1）—— = Q_B 合并产物
直接渲染 `actor_self.md`「我对你的了解」段的**干净、合并后**的偏好列表（`.acKnow`）。**这段的质量依赖 §7 的 Q_B**（写进段 + 合并，不再尾部堆流水账）。空态：`还在读你的口味——演几场我就懂了`。

### 生涯年表（v1）—— = 成长记
渲染 `actor_self.md`「成长记」段的时间线（`.acEntry` = 日期 + 人话理由 → 变化）。**成长记在这里是特性不是负担**——它是演员的"生涯大事记"，放这个 surface 就合理了（而**不再塞进每次生成的 prompt**，见 §7）。

### 擅长题材（v1.1）
各卡 `tags`/`topics` 聚合成标签（`.acSpec`，复用 `.tag` 视觉）：`悬疑 · 冷面 · 都市`。

### 角色名录 / 出演表（v1.1）
演过的角色 + 各演出量（`.acRole`）：`凛 · 42 轮`、`电子魅魔 · 18 轮`。像演员的作品年表。**不叫"搭档名录"**（同上：这些是角色/戏路，不是搭档）。

## 3. 亲密度阶梯（5 档）

| 级 | 分（默认阈值，env 可调） | 一句 |
|---|---|---|
| **初见** | < 15 | 刚认识，还在摸你的脾气 |
| **相识** | 15–39 | 演过几场，记住了你几样 |
| **搭档** | 40–99 | 有默契雏形，接得住你的球 |
| **默契** | 100–249 | 一个眼神就懂，越演越顺 |
| **知己** | ≥ 250 | 最懂你怎么玩的那个演员 |

阈值宽松、早期就有移动感（新用户一场戏就从初见→相识）。名字/阈值是 env 常量，二创可调。

## 4. 读路由契约 `GET /api/actor_card`

**纯聚合读，无写、无新事件**。server 端算好，reader 只渲染：

```json
{
  "name": "墨",
  "tagline": "能钻进任何角色里的人",
  "career": { "debut_days": 12, "productions": 4, "turns": 128, "words": 123000, "roles": 3 },
  "intimacy": {
    "level": "默契", "score": 176, "turns": 128, "log": 6,
    "next": "知己", "to_next": 74, "progress": 0.51
  },
  "knows": ["爱慢热的戏、回复别太长", "吃技术/物件的冷隐喻", "别替你的角色做决定"],
  "timeline": [
    { "date": "2026-06-29", "reason": "雨夜侦探那场你嫌铺太快", "change": "回复收短、慢热起势" }
  ],
  "specialties": ["悬疑", "冷面", "都市"],           // v1.1，v1 可空
  "roles_played": [{ "name": "凛", "turns": 42 }]     // v1.1，v1 可空
}
```
（`career.roles` = 戏路数；`intimacy.log` = 年表条数 L，驱动亲密度；`knows` = 「我对你的了解」合并档，只作展示不驱动亲密度。）

## 5. `ac*` 组件词汇表（在设计语言下）

新 surface 前缀 `ac*`（`liveware-frontend.md §4`）。**复用** `console.css` 的 token + 既有组件（`.pHead`/`.tag`/`.mark`/`.md`），只加下列：

| 类 | 用途 | 复用 |
|---|---|---|
| `.acPage` | 根：居中窄栏、warm paper、可滚 | `--bg` `--surface` |
| `.acHero` | 身份头：`✦ 墨 · 演员` + tagline | `.mark` `--brand` `--serif`(tagline) |
| `.acStats` / `.acStat` | 生涯数值格：大数 + 小标 | `--ink`(数) `--muted`(标) |
| `.acIntimacy` | 亲密度块：级名 + 一句 + 进度 | `--brand` |
| `.acProg` / `.acProgBar` | 进度槽 / 进度填充 | `--line2` / `--brand` |
| `.acKnows` / `.acKnow` | 我对你的了解列表 | `.knowList` 同款圆点 |
| `.acTimeline` / `.acEntry` | 生涯年表：日期 + 理由→变化 | `--muted`(日期) `--ink2`(正文) |

| `.acRoles` / `.acRole` | 角色名录 / 出演表（v1.1） | `--ink2` `--muted` |

**分区标题**统一用既有 `.pHead`（小字 uppercase muted），不新造。

## 6. 布局（单栏阅读，响应式）

**不是** master-detail（那是 console）——演员卡是**一张卡**：居中窄栏（复用 `.thread` 的 ~486px 阅读宽）、warm paper、留白大。数值用 `--sans`，tagline/年表理由可 `--serif` 添书页感。**无渐进披露**（这是展示 surface，控件少）。移动 = 同一单栏、收窄 padding，不需抽屉。顶栏复用 `#topbar` 形（`✦ 墨 · 演员` 居中）。

**边界态与实现注记**：
- **0 剧组**：出道 0 天、各值 0、亲密度「初见」；「我对你的了解」空态 `还在读你的口味——演几场我就懂了`；年表空态 `还没有生涯记录`。别显负数/NaN。
- **出道锚点**：`min(production.created_at)` 会在删光剧组后重置。v1 可接受（多数用户不删光）；若要更稳，v1.1 锚到 `actor_self` 首次落盘时间。
- **数值格式**：字数超 1 万显 `Y.Y 万`、否则显整数；天/轮/戏路都是整数，别漏 `round`。

## 7. 依赖的管道改动（Q_B / Q_C —— 让卡里的数据是"活的"）

演员卡是**展示层**；它展示得好不好，取决于这两个**技艺层管道**改动（本 spec 的前置依赖，落地时一起做）：

- **Q_B · 合并写入 + 注入瘦身**：
  - `ev_actor_grow` / `ev_reflect` 改为**写进「我对你的了解」段并合并**（精化替换、矛盾收敛、有界 ~10–15 条），**不再尾部堆**；同时在「成长记」记一条**审计条目**（日期 + 人话理由 → 变化）。→ 卡的「我对你的了解」是干净的、卡的「生涯年表」是审计流水。
  - `build_messages` **停止注入「成长记」**，只注 **底色 + 我对你的了解 + 签名**。→ 生成更精、不吃预算（治我 A/B 发现的注入膨胀）。**成长记从"每次 prompt 的负担"变成"演员卡里的资产"。**
- **Q_C · auto-reflect 服务端触发**：切 loadout 离开某剧组 / 达到 N 轮时，服务端自动 `reflect_on_play`（不靠墨自觉），蒸馏进「我对你的了解」+ 记一条年表。→ 卡自己会长，不等墨想起来。
