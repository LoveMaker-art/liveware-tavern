# 两个面：功能与特性（v1 设计起点）

> 承接 `../research/agent-liveware-opportunities.md`「RP 主舞台 = 混合」。这里把 chat 面与 liveware 面各自的功能/特性定下来。
> 状态：讨论中（2026-06-28 起）。每个「待定」是真问题。

## 0. 核心设计原则（canon，2026-06-28 用户定）

**继承酒馆用户的习惯与认知。** 不重造——ST 用户该有"家"的感觉。落到两条可执行铁律：

1. **常用功能 liveware 全覆盖**：日常高频的（角色卡、世界书查看/加条目、重生成/swipe/编辑、人设、开场白）一个不少，就在 UI 里。
2. **被简化的复杂功能，全部能通过和 agent 对话实现**：删进"设置海"的那些（采样参数、世界书触发机制、模板…）不是消失，是**换成"跟搭子说一句"**。例：不调温度滑块 → "回复短一点"；不编世界书递归规则 → "凛一提到水就紧张"。

**为什么只有我们能这么简化**：ST 没有 agent，只能把所有旋钮摊开（这正是它劝退的根源）；我们有真演员，复杂度可以移进对话——而且说一次就进**技艺/偏好层**长期记住（比 ST 手调 preset 更强）。**简化不是砍功能，是把功能从"旋钮"换成"对话"。**

---

## 修正过的两面分工（重要）

混合不是"所有 RP 都在控制台"。准确说：

- **chat 面 = 演员/搭子在身边**——既能像朋友一样自由聊，**也能随手试戏**（轻量、即兴、不建剧组）。角色扮演体验**不完全依赖 liveware**。
- **liveware 面 = 认真的剧组**——持久、分明、有独立故事线的深度演出，富面（沉浸对话 + 世界书 + 角色卡管理）都在这里。

一句话：**chat 是"随手玩 + 搭子关系 + 快速入口"，liveware 是"认真演 + 分明多线 + 沉浸"。**

---

## Chat 界面（IM）

### 1. 与搭子/演员自由交流
- 像朋友一样的自由聊天（出戏、闲聊、meta：「刚那段你自己觉得怎么样」）。
- 演员的跨剧组自我：记得你、懂你的口味（技艺/对你的了解层就在这个面体现）。

### 2. 随手试戏（chat 内轻量 RP）
- 演员能即兴接一个角色演一段，**不需要建剧组**——「演个高冷店长」→ 当场起戏。
- 用演员的技艺层（懂你），但**不挂剧组故事记忆**（即兴、短命、不污染任何正式剧组）。
- **晋升流（待定，建议做）**：一段试戏好玩 → 一键「**存成剧组 / 认真演**」→ 把这段搬进 liveware 成为持久剧组（带独立故事线）。这是连接两面最自然的动作。

### 3. liveware 的管理 / 启动入口
- 在 chat 里**添加世界书、角色卡**（粘贴/导入一张 Chub 卡 → 进你的库）。轻量、对话式的管理。
- **agent 主动丢入口**：演员主动甩一张可点的启动卡（「要不要试试这个场景？」）→ 点开直接把 liveware 拉进对应剧组。复用现有**活件卡**机制（`*.apps.clawling.io` 链接渲染成入口卡）。
- 定位：chat = **快速/对话式**管理 + 一键启动；liveware = **完整**库管理。两个入口不重叠（见下「管理入口分工」）。

## Liveware 界面（控制台）

### A. 贴近酒馆习惯，但适当简化
**保留**（让酒馆用户有家的感觉）：角色卡呈现、世界书条目、对话视图、角色面板、重生成/swipe（备选回复）、编辑消息、群聊/多角色（后续）。
**砍掉/收起**（治酒馆"设置海"的劝退）：采样参数、prompt 模板、instruct mode、API/key 配置、上下文 token 微操——**这些 agent 全接管**，用户不该看见。对齐设计声音「设置收敛」。

### B. 加强对话界面的沉静感
- 视觉留白、克制 chrome，让叙事**呼吸**（对齐「空气感」设计声音）。
- 叙事段落偏"阅读/小说"质感，入戏时 UI 家具最小化。
- **沉静 vs 功能密度的解法 = 渐进披露**：默认安静，控制项（重生成/编辑/角色面板/世界书）**按需出现**（点/悬停/上滑），不常驻挤占。
- 与酒馆"控制密布、信息过载"的 UI 形成反差——这是我们的体验差异点。

### C. 移动端 + 桌面端适配（liveware 是容器内 web UI，必须响应式）
- **桌面**（独立可缩放窗口）：可走 master-detail——剧组列表 / 对话 / 角色+世界书面板并排。
- **移动**（全屏 UrlViewerPage）：对话优先，堆叠/分页，拇指可达；剧组列表、角色面板、世界书走抽屉/分页切换。
- **一套 web UI 自适应**，桌面→手机 reflow。

## 跨两面的机制

- **试戏 → 剧组晋升**：chat 即兴 → 一键存成 liveware 剧组（带独立故事线）。
- **管理入口分工**：chat = 对话式快速添加 + agent 主动启动卡；liveware = 完整库管理（增删改、切换、编辑卡/世界书）。
- **两面靠同一份 agent state 保持一致**：演员技艺 + N 剧组状态都在 agent 侧；chat 与 liveware 是同一 state 的两个视图，天然同步。

## 线框落定的具体决定（2026-06-28，已出移动+桌面两版）

**Chat 面**：
- 联系人 = **演员本人「墨」**（一个固定身份），不是某角色——印证"IM 联系人保持演员身份"。桌面上它就和资讯日报/其他 agent 并列在 master-detail 列表里。
- 自由聊 + 随手试戏**同处一条线**；试戏回复下挂一枚 `存成剧组` chip = 晋升动作。
- agent 主动丢的启动卡 = 复用**活件卡**（✦ + 剧组名 + 「点击进入剧组」）。
- 输入框 `+` = 加卡/世界书的管理入口。

**Liveware 控制台**：
- 顶部 ✦ 活件 chrome（品牌标 + 剧组名居中），与启动卡/digest 同一个 ✦。
- **沉静感**落成：叙事用 serif 斜体、留白大；角色台词是干净文本块（不是社交气泡）；用户输入**后退**（muted、右对齐、弱化）——和 chat 面的气泡热闹刻意反差。
- **渐进披露**落成：重生成/编辑/swipe 控制条**ghost 态**（低透明），点消息才显，不常驻。
- **ST "设置海"全不可见**：无采样参数/prompt 模板/API 配置——agent 全接管。
- 桌面 = master-detail（剧组列表 · 沉浸对话 · 角色/世界书面板）；移动 = 全屏舞台 + 顶部 `← 墨` 回到搭子（移动动线）。

## ST 功能覆盖契约（每个功能都有家，不许漏）

落点三档：**liveware**（常用，UI 覆盖）｜**对话**（复杂，跟 agent 说，说一次进技艺层）｜**自动**（推理底层，用户永不见）。

| 酒馆功能 | 落点 | 怎么实现 / v1 |
|---|---|---|
| 角色卡基本（名/简介/性格/场景/开场白 first_mes） | liveware | 导入 + 查看 + 轻量改。**v1** |
| 备选开场白 alternate greetings | liveware | swipe 换开场。**v1** |
| 用户人设 persona | liveware | 简单卡：你在戏里是谁。**v1** |
| 世界书 查看 + 加条目 | liveware | 列表 + 加一条（关键词 + 内容）。**v1** |
| 重生成 / swipe 备选 / 编辑消息 | liveware | 渐进披露控制条。**v1** |
| 角色卡高级（system_prompt / post_history / 对话示例） | 对话 | "更高冷点 / 别用现代词" → agent 内化进演法 |
| 世界书高级（递归/扫描深度/常开蓝灯/二级关键词/位置/order/token 预算） | 对话 | "凛一提到水就紧张" → agent 自己拿捏触发，不暴露机制 |
| 采样参数（温度/重复惩罚/top-p… 21 项） | 对话 | "回复短点 / 别重复 / 更有文采" → 说一次进技艺层长期记住 |
| 切模型救场 | 对话 | "换个写法 / 认真点" → agent 侧切，用户不碰 key |
| 长期记忆（summarize / 向量 / 作者注释） | 自动 + 对话 | 两层记忆自动跑；"记住我们约了周五" 显式补 |
| instruct / context 模板 | 自动 | agent 拼 prompt，用户永不见 |
| API / 后端 / key 配置 | 自动（删） | agent 接管推理，整块消失——**酒馆最大摩擦点** |
| Quick Replies / STscript 自动化 | 对话 + 活件卡 | agent 本身 + 主动丢的启动卡替代 |
| 群聊多角色（talkativeness / 回复顺序） | liveware + 对话 | 选角进剧组（UI）+ "让凛多说话"（对话）。**后续** |
| 出图 / TTS / 立绘 / Live2D | liveware | 富面开关，复杂配置走对话。**后续** |

**读法**：标 v1 的 = 第一版 liveware UI 就有；「对话」档从第一天起就可用（不需要 UI，跟搭子说就行）；「后续」= 富面留迭代。**没有任何一格是"消失了没法做"。**

## 浮出来的设计决策 / 待定

- [ ] **试戏→剧组晋升流**的具体形态（一键？演员提议？）—— 建议做，是两面的黏合剂。
- [ ] **移动端的"两面"体感**：桌面上 liveware 是独立窗口、能和 IM 并排；移动上 liveware 是全屏页，IM↔liveware 是**前后导航**而非并排。两面在移动端是"切换"不是"并存"——要不要为移动设计专门的来回动线？
- [ ] **沉静感 vs 酒馆用户要的控制**（重生成/swipe/编辑/角色面板）——渐进披露能不能既安静又不丢功能，待原型验。
- [ ] **v1 富面边界**：v1 控制台 = 基础 per-剧组对话 + 切换器 + 卡/世界书最简管理；立绘/状态栏/世界书递归浏览器等富面留后续。
- [ ] **世界书/角色卡库的呈现与简化编辑**：库住 agent 侧，控制台怎么展示、改到什么粒度。
